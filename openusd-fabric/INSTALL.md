# openusd-fabric per-host install

This is the operational reference for hooking the V-Sekai USD schema
into Blender, Godot (via `idtx-flow`), and Unity. The schema design
lives in CHI-250; this page is purely "how to make it load".

## Repo layout

```
openusd-fabric/
├── schema/                       # Codeless USD schema bundle
│   ├── v_sekai_schema.usda       # Source (edit this)
│   ├── generatedSchema.usda      # Generated registration source
│   └── plugInfo.json             # Plugin manifest USD reads at warmup
├── blender/
│   ├── pre_export_mirror.py      # RNA → id_property mirror
│   └── post_export_hook.py       # userProperties → schema API + spring synthesis
├── lean/                         # Lake project — proof-pinned artifacts
│   ├── Fabric/                   # Specs (HumanoidBones, ScssMToon, ...)
│   └── EmitArtifacts.lean        # Lake exe that emits maps/*.json
├── maps/                         # Emitted by Lake exe, committed
│   ├── humanoid_bones_map.json
│   ├── scss_mtoon_map.json
│   └── springbone_fields_map.json
└── tests/
    ├── blender/                  # pytest for the two Blender hooks
    └── fixtures/                 # USD round-trip corpus
```

## The PXR_PLUGINPATH_NAME contract

**All three hosts** need `openusd-fabric/schema/` on
`PXR_PLUGINPATH_NAME` **before** any `UsdStage::Open` call. Setting
it later is a silent no-op — USD warms its plugin registry once, at
first stage operation in the process.

Without this, `prim.HasAPI("VSekaiSpringBoneAPI")` returns false
from C++ even when the apiSchemas metadata is present. (Python is
more lenient.)

### How to verify it's set

```bash
# Bash (POSIX paths — won't work on Windows USD CLI tools)
echo $PXR_PLUGINPATH_NAME

# PowerShell (use this on Windows; idtxcli requires drive-letter paths)
$env:PXR_PLUGINPATH_NAME
```

If you see the schema dir in the output AND your USD invocation
runs in the same process, you're good.

## Codeless schema: why `generatedSchema.usda` is committed

Codeless schemas (`skipCodeGeneration = true`) ship as a pair:

* `v_sekai_schema.usda` — the source. Authored by humans. Defines
  the API classes, attribute types, default values, docstrings.
* `generatedSchema.usda` — the runtime registration source. Read by
  USD at plugin warmup to populate the schema registry. Required
  for `HasAPI`, `ApplyAPI`, default-value resolution.

`generatedSchema.usda` is **not** a build artifact you regenerate
on every build. It's a checked-in interchange format, regenerated
only when you change the source. The convention in many USD
codebases is to gitignore it; for codeless schemas, that's wrong
— without it, downstream consumers can't load your schemas.

### Regenerating after a schema edit

From `openusd-fabric/schema/`:

```powershell
$env:PXR_PLUGINPATH_NAME = "<repo>\thirdparty\openusd-25.11-withPython\lib\usd"
$env:PYTHONPATH = "<repo>\thirdparty\openusd-25.11-withPython\lib\python"
$env:PATH = "<repo>\thirdparty\openusd-25.11-withPython\lib;" + `
            "<repo>\thirdparty\openusd-25.11-withPython\bin;" + $env:PATH
python ..\..\thirdparty\openusd-25.11-withPython\bin\usdGenSchema `
    v_sekai_schema.usda .
```

Hand-trim the resulting `plugInfo.json`: `usdGenSchema` emits one
entry per schema class PLUS one autoGenerated alias per class
prefixed with `libraryPrefix` (yields `VSekaiVSekaiMToonAPI` etc).
The aliases are harmless but noisy; drop them and keep only the
four canonical type names.

## Blender (4.x)

### Install the addon dependency

```
Edit → Preferences → Add-ons → Install → vrm-addon-for-blender
```

Then enable it (`bl_ext.blender_org.vrm`). The pre-export mirror
reads its RNA tree.

### Set PXR_PLUGINPATH_NAME for Blender's bundled USD

Blender ships its own USD build under
`<blender>/<version>/python/lib/usd/`. You can either:

* **Per-launch (developer loop):**
  ```powershell
  $env:PXR_PLUGINPATH_NAME = "<repo>\openusd-fabric\schema"
  & "C:\Program Files\Blender Foundation\Blender 4.x\blender.exe"
  ```
* **Persistent (CI / users):** drop a `plugInfo.json` symlink or
  copy the `schema/` directory next to Blender's bundled
  `usd/resources/` so USD auto-discovers it on warmup.

### Run the pre-export mirror, then the post-export hook

```powershell
# 1. Inside the open .blend, mirror RNA → id_properties
blender --background <project.blend> `
    --python openusd-fabric\blender\pre_export_mirror.py

# 2. Use Blender's USD export
#    (File → Export → Universal Scene Description, or
#    bpy.ops.wm.usd_export(filepath="out.usda", export_animation=False))

# 3. Stamp V-Sekai API schemas + synthesise spring prims
python openusd-fabric\blender\post_export_hook.py --in out.usda
```

## Godot (via `idtx-flow` / `idtxcli`)

### CLI smoke test

```powershell
$env:PXR_PLUGINPATH_NAME = "<repo>\thirdparty\openusd-25.11\lib\usd;" +
                          "<repo>\openusd-fabric\schema"
.\build\idtxcli\idtxcli.exe usd-to-vrm  input.usda output.vrm
.\build\idtxcli\idtxcli.exe vrm-to-usd  input.vrm  output.usda
.\build\idtxcli\idtxcli.exe usd-to-usd  input.usda output.usda
.\build\idtxcli\idtxcli.exe version
```

The smoke harness (`tools/idtxcli/smoke_test.ps1`) does this for
every fixture in `openusd-fabric/tests/fixtures/`. PowerShell only
— bash POSIX paths (`/c/Users/...`) break Windows USD CLI tools.

### GDExtension consumers

`idtx-flow` registers the schema dir during its GDExtension entry
point (`scons/gdextension.py` → `idtx_core_module_init`). End users
don't need to set anything in Godot's project settings.

## Unity (VRM consumption only, no VRChat)

Scope cut on 2026-05-23: Unity work targets VRM 1.0 consumption,
not VRChat / PhysBone authoring. The `unity/IdtxCore/` UPM package
gives a Unity 2022.3.x project a `[InitializeOnLoad]` hook
(`Editor/IdtxCoreLoader.cs`) that prepends the schema dir to
`PXR_PLUGINPATH_NAME` before any USD API call.

For the package's bundled `idtx_core.dll`, the loader looks for
`Packages/com.v-sekai.idtx-core/Schema/` (a symlink or copy of
`openusd-fabric/schema/`). Either way, the user never sets the env
var manually.

### Importing

* `Editor/IdtxUsdImporter.cs` is a `ScriptedImporter` that consumes
  `.usda` files and emits a Unity GameObject hierarchy with a
  populated `IdtxAvatar` component.
* `Bridge/UnityAvatarBridge.cs` does the bidirectional bridge.

There is no UniVRM10 prefab emission step — that was the
VRChat-leg deliverable on CHI-251 and is out of scope.

## CI gates

Every PR must pass:

1. **Blender pytest** (`openusd-fabric/tests/blender/`) — 20 cases,
   bpy mocked, no real Blender needed.
2. **Lake build** (`openusd-fabric/lean/`) — `native_decide` proofs
   pinning every emitted artifact; non-zero exit on drift.
3. **`maps/*.json` bytes match emitted output** —
   `openusd-fabric/tests/lean/test_lake_builds.py`.
4. **idtxcli smoke test** (`tools/idtxcli/smoke_test.ps1`) —
   USD ↔ VRM round-trip on every fixture; `usdchecker --arkit=false`
   gate; `usdcat --flatten` diff line counts reported as
   informational.

## Common gotchas

* **"Failed to open schema layer at path '.../generatedSchema.usda'"**
  — `generatedSchema.usda` isn't in your schema dir. Either commit
  it (preferred) or run `usdGenSchema` per the regen instructions
  above. The schema warnings cascade into "No prim spec exists at
  path '/VSekai*API'" follow-on warnings; ignore those, they're
  the same root cause.
* **`prim.HasAPI(...)` works in Python but returns false in C++** —
  the schema dir isn't on `PXR_PLUGINPATH_NAME` at process start.
  Python's `HasAPI` is more lenient and reads apiSchemas metadata
  directly; C++ consults the schema registry.
* **bash on Windows can't run idtxcli with `/c/Users/...` paths in
  env vars** — USD expects drive-letter paths. Use PowerShell, or
  convert via `cygpath -w "$(pwd)"`.
* **`v_sekai:springBone:joints` is `int[]`, not `token[]`** — joint
  indices into the Skeleton's `joints` token array, not bone-name
  strings. Same for `colliders` (walk-order indices) and
  `collider:attachedBone` (single int). Bone names and collider
  names ride alongside on custom `*Names` / `*Name` attrs for
  debugging.
